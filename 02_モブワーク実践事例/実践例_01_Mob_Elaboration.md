# モブワーク実践ガイド 1: Mob Elaboration実践例

## シナリオ
「クロスセル向け推薦エンジンを開発する」というビジネス意図をIntent化し、ユーザーストーリー・Unit・Boltへ分解するセッション

## 所要時間
4～6時間（午前1セッション 3時間 + 午後1セッション 3時間で実施も可能）

## 参加者
- **ファシリテータ**（Scrumマスター or テックリード）
- **AI（アシスタント）**（Claude, ChatGPT等のエージェント機能）
- **プロダクトオーナー**
- **開発者 2～3名**
- **QA/テスター**
- **必要に応じてドメイン専門家**

## セットアップ

### ステップ0: プリセッション準備（セッション前日）

以下をPOから確認し、AI向けプロンプトに含める文脈を準備する。

```markdown
## Intent背景（ファシリテータが事前確認）
- ビジネス目標: 顧客1人当たりの平均注文額を15%向上
- 対象期間: 3ヶ月のパイロット
- 対象ユーザー: 既存顧客（セグメント未定）
- 主要な成功指標: クリック率、コンバージョン率、API応答時間
- 制約条件: 既存決済システムとの互換性必須、GDPR対応必須
```

### ステップ1: セッション開始（15分）

**ファシリテータのタスク:**
1. 参加者全員に役割と進め方を説明
2. 結果成果物（ユーザーストーリーファイル、Unitマップ、Boltプラン）の格納先を確認
3. 「質問は遠慮なく」の心理的安全性を強調

**サンプルセッション開始スピーチ:**

```
本日は、POからの意図「クロスセル向け推薦エンジン」を
ユーザーストーリー・Unit・Bolt へ分解します。

AIが最初の案を提案し、
私たちはそれを現実的に調整・充実させます。

この4～6時間で、開発チーム全体が同じ理解を持ち、
来週から即座に着手できる状態を目指します。
```

---

## フェーズ1: Intent明確化（60分）

### 1-1. AIへの初期プロンプト

```
あなたは経験豊富なプロダクト分析官です。
以下のビジネスintentを分析し、
明確化に必要な質問を5～7個、番号付きリストで提示してください。

Intent: クロスセル向け推薦エンジンを開発する

既知の制約:
- GDPR対応必須
- 既存決済システム互換性必須
- 3ヶ月のパイロット期間

質問後、提示した各質問に対し、
以下の回答ケースを想定して、
「このような回答が得られた場合、分解方法はどう変わるか」
を簡潔に説明してください。
```

### 1-2. AI回答例

```
【AIからの質問例】

1. 推薦対象: どの商品カテゴリが対象か。
   食品のみ、家電、クロスカテゴリか。

2. ユーザーセグメント: 
   全既存顧客か、特定セグメント（例: VIP）か。

3. 推薦トリガー: 
   購入直後、ログイン時、ホーム画面か。

4. データソース: 
   内部購買履歴、外部DMP、LLMベースの分析か。

5. 個人化レベル: 
   単純な「人気商品」か、個人嗜好学習か。

6. スケール要件: 
   日次レコメンド100万件、リアルタイムAPI、
   どの粒度か。

7. リスク許容度: 
   不正確推薦による顧客不満をどこまで許容か。
```

### 1-3. ファシリテータのモデレーション

```markdown
【ファシリテータが実施】

Q1（推薦対象）への回答:
「今回のパイロットでは、全商品を対象にします。
 ただしコスメと食品の相乗効果を重視してテストしたいです」

→ AIへ追問:
「コスメと食品の相乗効果とは、具体的にどのシナリオか。
 例えば『スキンケア購入者に対し食品の栄養系を推薦』の意？」

PO: 「そうです」

→ ファシリテータが参加者に確認:
「開発側として、この相乗関係を
 シスム上でどう表現しますか？（データ構造的に）」

開発者A: 「SKUメタデータに相乗関係スコアを持たせ、
        レコメンドエンジンがそれを参照する方が
        運用しやすいと思います」

→ ファシリテータ: 「それを設計のポイントとして記録しておきます」
```

### 1-4. Intent最終定義書の作成

セッション終了時に、以下テンプレートで成果物を残す。

```markdown
## Final Intent Definition

**意図:** クロスセル向けリアルタイム推薦エンジン

**ビジネス成果:**
- 顧客1人当たり平均注文額 +15%
- 推薦クリック率 目標 8% 以上
- 推薦経由コンバージョン率 目標 2.5% 以上

**対象ユーザー:**
- 既存顧客全体（パイロット期間）

**推薦内容:**
- 購入商品のコスメ・食品カテゴリの相乗効果を学習
- 例: スキンケア購入者→美容向け食品推薦

**トリガータイミング:**
- 購入完了直後、その他トランザクション確認画面

**技術要件:**
- API レスポンス < 200ms
- 精度: リコール 60% 以上
- GDPR対応（個人データ保護、削除権実装）
- 既存決済API互換性維持

**リスク:**
- 低精度による顧客体験低下
- 推薦ロジック複雑化による運用負担
- スケーリング時の遅延

**制約:**
- パイロット期間 3ヶ月
- MVPスコープから開始
```

## フェーズ2: ユーザーストーリー生成（90分）

### 2-1. AIへのプロンプト

```
先ほど確定したIntent定義に基づいて、
このシステムに必要なユーザーストーリーを列挙し、
各ストーリーに対し、受け入れ基準を記述してください。

形式:
[US-001] ユーザーストーリータイトル
- アクター: （例: 既存顧客）
- ストーリー: As a [アクター], I want [機能],
             so that [価値]
- 受け入れ基準:
  - [ ] 基準1
  - [ ] 基準2
  - [ ] 基準3

その後、リスクまたは不確実性が高いストーリーを
「検討が必要」とマークしてください。
```

### 2-2. AI出力例

```
## ユーザーストーリー初期案

[US-001] 購入完了後、推薦商品をメール受け取る
- アクター: 既存顧客
- ストーリー: As a 既存顧客, I want 購入数分後に
             パーソナライズ推薦メールを受け取り,
             so that 自分の嗜好にあった新商品を
             発見できる
- 受け入れ基準:
  - [ ] メール配信遅延 < 5分
  - [ ] 推薦商品3～5個を表示
  - [ ] クリック可能なリンク含む
  - [ ] テスト顧客で正常送信確認
- 検討が必要: GDPR同意管理（既存同意との整合）

[US-002] 推薦ロジックをコンソールで設定できる
- アクター: マーケティングマネージャー
- ストーリー: As an マーケティングマネージャー,
             I want 相乗効果マトリクスを管理画面で
             調整し,
             so that 季節や在庫に応じた
             推薦ポリシーを変更できる
- 受け入れ基準:
  - [ ] CSVアップロード機能実装
  - [ ] 変更が30秒以内に反映
  - [ ] 変更履歴ログ記録
- 検討が必要: 権限管理（誰が変更OK？）
- 検討が必要: A/Bテスト機構との連携

...（以下複数ストーリー）
```

### 2-3. ファシリテータによる検討と調整（30分）

```markdown
【参加者による共同レビュー】

ファシリテータ: 「[US-001]メール配信について、
             誰か追加の懸念や質問がありますか？」

QA: 「受け入れ基準の『推薦商品3～5個』は、
    ユーザーが多すぎると感じる場合の対応は？」

PO: 「まずはA/Bテストで最適数を探しましょう。
    初期は3個固定でいきます」

開発者B: 「メール本文のテンプレート管理はどこでやります？
        マーケティング用CMS、それとも当システム内？」

PO: 「既存メール配信システム(Salesforce)と連携します」

ファシリテータ: 「その連携仕様を新しいストーリーにすべきですね。
           [US-001a] 'Salesforce連携'として追加します」
```

### 2-4. ユーザーストーリー確定リスト

セッション後、各ストーリーをドキュメント化し、
チーム全員がアクセス可能な場所に保存。

## フェーズ3: Unit分解（60分）

### 3-1. AIへのプロンプト

```
上記ユーザーストーリー群を、
独立して構築・テスト・デプロイ可能な
「Unit」へグループ化してください。

各Unitは以下を満たす必要があります:
- 高凝集：1つの業務価値を提供
- 疎結合：他Unitへの依存を最小化
- 実装可能：1～2週間で完成
- テスト可能：自動テスト基盤を含む

形式:
[Unit-A] Unitの名称
- 含まれるストーリー: [US-001], [US-002], ...
- 提供価値: 一文で
- 依存Unit: （あれば）
- 推定NFR:
  - パフォーマンス
  - セキュリティ
  - スケーラビリティ
```

### 3-2. AI出力例

```
## Unit初期案

[Unit-A] メール推薦配信基盤
- 含まれるストーリー: [US-001], [US-001a]
- 提供価値: 購入後パーソナライズメール配信機構
- 依存Unit: [Unit-B] 推薦ロジック（データ入力）
- NFR:
  - パフォーマンス: 配信遅延 < 5分
  - スケーラビリティ: 日次10万件以上対応
  - 監視: 配信失敗率 < 0.1%

[Unit-B] 推薦ロジック・マネジメント
- 含まれるストーリー: [US-002], [US-003]
- 提供価値: 相乗効果マトリクスの定義・管理
- 依存Unit: なし（独立）
- NFR:
  - パフォーマンス: 変更反映 < 30秒
  - セキュリティ: アクセス制御必須
  - 可用性: 99% SLA

[Unit-C] レコメンドエンジンコア
- 含まれるストーリー: [US-004], [US-005]
- 提供価値: 購買→推薦アルゴリズム実装
- 依存Unit: [Unit-B] マトリクスデータ
- NFR:
  - パフォーマンス: API < 200ms
  - 精度: リコール 60% 以上
  - スケーラビリティ: リアルタイム処理
```

### 3-3. ファシリテータのモデレーション

```markdown
ファシリテータ: 「Unit-Cのパフォーマンス 200ms というのは、
             他システムのSLAと整合してますか？」

開発者A: 「既存ホーム画面API全体が1秒以内なので、
        200msはきつい。250～300msで交渉したい」

PO: 「顧客体験の遅延は許容されないので、
   200msまで詰めてほしい。
   ただしその代わり、パイロット期間は推薦対象を
   ホット商品（在庫豊富）に限定でもいい」

ファシリテータ: 「つまり精度より速度を優先する期間ってことですね。
           それをUnit-Cの受け入れ基準に反映します」
```

### 3-4. Unit確定図

```
Unit Dependencies（Unitマップ）:

[Unit-B] Recommendation Logic Management
         ↓
[Unit-C] Recommendation Engine Core
         ↓
[Unit-A] Email Delivery Pipeline

Parallel Development:
- Unit-B: Week 1-2
- Unit-C: Week 2-3 （Unit-Bの後）
- Unit-A: Week 3-4

Timeline: 全体 4週間（Sprint 1-2）
```

## フェーズ4: Bolt計画（30分）

### 4-1. Bolt定義

各Unitを、さらに時間～日単位で細分化する。

```
1つの Bolt = 1日以内で検証可能な作業

例：
Bolt-A1: Unit-Aのデータモデル定義＆ローカルDB構築
Bolt-A2: メール配信ロジックの実装（Salesforce連携なし）
Bolt-A3: Salesforce連携テスト
Bolt-A4: 本番環境への段階的ロールアウト
```

### 4-2. AI が Bolt 案を提示

```
あなたは経験豊富なSprint計画者です。
上記Unit群を、以下Boltプランに分解してください。

各Boltは:
- 1日以内に実装＋テスト完了
- 1つの機能または責務に限定
- 明確なDone定義を持つ

形式:
[Bolt-XY] タイトル
- 実装内容: 具体的な成果物
- 担当（提案）: 誰が適任か
- 依存Bolt: 前提となるBolt
- Done定義:
  - [ ] コード完成
  - [ ] ユニットテスト pass
  - [ ] コードレビュー完了
  - [ ] インテグレーションテスト pass
```

### 4-3. Bolt スケジュール確定

```markdown
Week 1:
- Bolt-B1: 推薦マトリクスデータモデル定義
- Bolt-B2: 管理画面のフロント実装（ダミーバックエンド）

Week 2:
- Bolt-C1: レコメンドAPI骨組み実装
- Bolt-C2: アルゴリズムロジック実装（テスト環境）
- Bolt-C3: パフォーマンステスト＆最適化

Week 3:
- Bolt-A1: メール配信テンプレート設計
- Bolt-A2: メール配信タスク実装
- Bolt-A3: Salesforce連携実装

Week 4:
- 統合テスト
- ステージング検証
- 本番デプロイ準備

→ このプランをチーム全員で共有し、
   次週月曜からBolt実行開始
```

## よくある失敗パターン

- Intent明確化を短縮しすぎて、後で分割/修正が増える
- AIの提案を無批判に採用
- ストーリー間の優先度をつけ忘れ、開発順序が曖昧
- Unit依存を見落とし、平列開発予定で実際は直列化
- Boltスコープが大きすぎて、1日以内完結しない
