# AI駆動開発ライフサイクル（AI-DLC）メソッド定義（日本語訳）

- 原文: https://prod.d13rzhkk8cj2z0.amplifyapp.com/
- 著者: Raja SP（Amazon Web Services）

## I. 背景（Context）

ソフトウェア工学の進化は、低レベルで差別化されない作業を抽象化し、開発者がより複雑な問題解決に集中できるようにする継続的な探求でした。初期の機械語から高水準プログラミング言語、さらにAPIやライブラリの採用に至るまで、各段階で開発者の生産性は大きく向上しました。

現在、大規模言語モデル（LLM）の統合によって、コード生成・バグ検出・テスト生成といった作業を自然言語の対話で行えるようになり、ソフトウェアの作り方は大きく変わりました。これは、AIが粒度の細かい個別作業を強化する「AI支援（AI-Assisted）時代」を示しています。

さらにAIは、コード生成を超えて、要件詳細化、計画立案、タスク分解、設計、開発者とのリアルタイム協業へと適用範囲を広げています。これは、AIが開発プロセスを積極的にオーケストレーションする「AI駆動（AI-Driven）時代」の始まりです。

しかし、既存の開発手法は人間主導かつ長いサイクルを前提として設計されており、AIの速度・柔軟性・高度な能力（例: エージェント性）とは十分に整合していません。手動ワークフローや硬直的な役割定義への依存により、AIの力を最大化できないのです。既存手法にAIを後付けするだけでは、AIの可能性を制限し、旧来の非効率を温存します。

AIの変革力を最大化するには、SDLC手法そのものを再構想する必要があります。その再構想では、AIを中心的な協働者として据え、ワークフロー・役割・反復サイクルを再設計し、高速な意思決定、シームレスな実行、継続的適応を実現しなければなりません。

本稿は、AI能力を全面的に統合するために再設計されたAIネイティブな方法論「AI-Driven Development Lifecycle（AI-DLC）」を提示し、その定義を与えます。

## II. 主要原則（Key Principles）

本章の原則は、AI-DLCのフェーズ、役割、成果物、儀式（ritual）を定義する土台です。

### 1. 後付けではなく再構想する

既存手法（SDLCやAgile/Scrum）にAIを後付けするのではなく、手法自体を再構想する。従来手法は週〜月単位の長い反復を前提としており、デイリースタンドアップやレトロスペクティブといった儀式を生んだ。対してAIを適切に活用すると反復は時間〜日単位に短縮され、継続的・リアルタイムな検証とフィードバックが必要となる。

たとえばAIがタスク難易度の境界を小さくするなら、ストーリーポイント見積りは従来ほど重要か。ベロシティよりビジネス価値を主指標にすべきではないか。AIは計画、分解、要件分析、設計技法適用（例: ドメインモデリング）まで自動化しつつあり、意図からコードまでの段階数を短縮する。ゆえに、既存手法の延長ではなく、第一原理思考による再設計が必要である。

### 2. 会話の向きを反転する

AI-DLCでは、人間がAIに依頼して会話を始めるのではなく、AIが会話を開始・主導する。AIは高レベル意図（例: 新しい業務機能の実装）を実行可能タスクへ分解し、推奨案とトレードオフを提示する。人間は重要局面で承認・選択・意思決定を行う。

これにより、開発者は高価値な判断に集中し、AIは計画・分解・自動化を担う。人間の関与は監督、リスク低減、戦略整合に集約され、速度と品質を同時に高める。

（比喩）Googleマップのように、人間は目的地（意図）を設定し、システム（AI）が段階的経路（分解・推奨）を提示する。人間は道中で監督し必要に応じて介入する。

### 3. 設計技法を中核に統合する

Scrum/Kanbanは設計技法（例: DDD）を枠外に置き、各チーム任せにしがちで、結果として品質の空白領域が生まれた。AI-DLCでは設計技法を中核に内蔵する。

DDD版、BDD版、TDD版などのAI-DLCバリエーションを持ち得るが、本稿はDDD版を扱う。DDD原則でシステムを適切な境界づけられたコンテキストへ分割し、並列に高速構築する。AIが計画と分解でこれら技法を内在的に適用し、開発者は検証・調整に注力する。これが時間〜日単位反復と品質維持（「より良いシステムをより速く」）を両立させる鍵である。

### 4. AI能力と整合する

本稿はAIの将来に楽観的だが、現状の限界も認識する。現行AIは高レベル意図を自律的に安全・解釈可能な実行コードへ確実変換する段階にはない。一方で、開発者が知的作業の大半を担いAIが補助するだけのAI支援モデルでは、潜在力を十分引き出せない。

AI-DLCは、現行AIの能力と限界を踏まえ、人間関与とのバランスを取るAI駆動モデルを採る。最終責任は開発者が持ち、検証・判断・監督を実施する。

### 5. 複雑システム構築を対象とする

AI-DLCは、継続的な機能適応、高いアーキテクチャ複雑性、多数のトレードオフ管理、スケーラビリティ、統合、カスタマイズを要するシステムを対象とする。通常は複数チームが大規模/規制環境で協調して開発するケースである。

トレードオフ管理がほぼ不要な単純システムや非開発者中心の開発は、AI-DLCの主対象外であり、ローコード/ノーコードが適する。

### 6. 人間との共生を高める要素は残す

再構想に際しても、人間の検証やリスク低減に不可欠な成果物・接点は保持する。たとえばユーザーストーリーは、人間とAIの認識を揃える契約として有効であるため保持する。リスク登録簿（Risk Register）も、AI生成計画/コードを組織のリスク枠組みに適合させるため保持する。

保持要素はリアルタイム利用向けに最適化し、高速反復と安全性を両立する。

### 7. 馴染みある用語で移行を容易にする

新手法は長期研修を前提とせず、既存実務者が1日で実践に入れることを目標にする。連想学習による導入のため、旧手法の用語関係は維持しつつ用語を現代化する。

例: ScrumのSprintは反復サイクルだが、AI-DLCでは時間〜日単位の連続反復となるため、Sprintを意図的にBoltへ改称する。Boltは短く高密度な反復を強調する。

### 8. 責務を統合し効率化する

AIの分解・意思決定能力により、開発者はインフラ、フロントエンド、バックエンド、DevOps、セキュリティといった専門分断を越えて動けるようになる。これにより、多数の専門ロール依存を減らし、開発を簡素化する。

ただしプロダクトオーナーと開発者は中核ロールとして残り、監督・検証・戦略判断を担う。方法定義では第一原理に基づき、ロール数を最小化し、本当に必要な場合のみ追加する。

### 9. 段階を最小化し、フローを最大化する

自動化と責務統合によりハンドオフを減らし、連続的な反復フローを実現する。一方、AI生成コードが将来変更しにくい「クイックセメント化」しないよう、人間検証と意思決定は必須である。

AI-DLCは、重要な判断点に限定した最小十分なフェーズを設ける。これらの検証は損失関数のように働き、下流で膨らむ無駄を早期に除去する。

### 10. 固定的・教条的ワークフローを持たない

AI-DLCは、新規開発、リファクタ、欠陥修正、マイクロサービス拡張などに対して固定ワークフローを処方しない。代わりにAI Firstで、経路意図に応じたLevel 1 PlanをAIが提案し、人間が対話的に検証・調整する。同様にLevel 2（サブタスク）以下へ再帰分解する。

実行レベルではAIが実装し、人間が結果を検証・妥当化する。この柔軟性により、AI能力進化に追随しつつ、重要意思決定の人間統制を維持する。

## III. コアフレームワーク（Core Framework）

### 1. 成果物（Artifacts）

- Intent: 何を達成するかを示す高レベル目的文。ビジネス目標、機能、技術成果（例: 性能スケーリング）を含む。AI分解の起点となる。
- Unit: Intentから導出される、測定可能価値を提供する自己完結な作業要素。DDDのサブドメインやScrumのエピックに相当。ユーザーストーリー群で機能スコープを表す。AIがIntent→Unit分解を行い、開発者/POが検証・改善する。Unitは疎結合で自律開発・独立デプロイが可能。
- Bolt: Unitまたはそのタスク集合を高速実装する最小反復。Sprint相当だが、周期は週ではなく時間〜日。1つのUnitは複数Boltで実行でき、並列/直列いずれも可能。AIがBolt計画を提案し、開発者/POが検証する。
- Domain Design: Unitの中核業務ロジックをインフラ非依存でモデル化した成果物。AI-DLC第1版ではDDDに基づき、集約、値オブジェクト、エンティティ、ドメインイベント、リポジトリ、ファクトリ等を設計。
- Logical Design: Domain Designを非機能要件に適合させるため、適切なアーキテクチャパターン（例: CQRS、Circuit Breaker）を適用した設計。AIはADR（Architecture Decision Record）を作成し、開発者が検証。
- Code and Unit Tests: Logical Designに基づき、AIが適切なAWSサービス/構成を選びつつコードと単体テストを生成。AIエージェントが単体テスト実行・結果分析・修正提案を行い、開発者が判断。
- Deployment Units: 実行可能コード（例: Kubernetes向けコンテナ、AWS Lambda）、設定（例: Helm Chart）、IaC（例: Terraform/CloudFormation）を含む運用成果物。機能受け入れ、セキュリティ、NFR、運用リスクで厳格検証される。AIが各種テスト（機能・静的/動的セキュリティ・負荷）を生成し、人間がシナリオ調整後にAIが実行・分析し、失敗要因をコード/設定/依存へ相関づける。

### 2. フェーズと儀式（Phases & Rituals）

#### Inception Phase

Intentを捉え、開発用Unitへ翻訳する。中核儀式はMob Elaboration（共同要件精緻化・分解）で、同室・共有画面・ファシリテータ主導で実施する。

- AIが、Intentをユーザーストーリー、受け入れ基準、Unitへ初期分解
- PO、開発者、QA、関係者（mob）がAI成果物を共同でレビュー/調整
- 出力は以下を含む
  - PRFAQ
  - ユーザーストーリー
  - NFR定義
  - リスク記述（組織のRisk Registerと対応）
  - ビジネス意図にトレース可能な測定基準
  - Unit実装のための推奨Bolt

Mob Elaborationは、通常数週間〜数か月の直列作業を数時間へ圧縮し、チーム内およびAIとの高い整合を作る。

#### Construction Phase

Inceptionで定義したUnitを、テスト済みで運用可能なDeployment Unitへ反復的に変換する。

- Domain Design: AIが業務ロジックを技術要素から独立してモデル化
- Logical Design: NFR適用とクラウド設計パターン適用
- Code Generation: AIがAWSサービスへマッピングして詳細コード生成
- Automated Testing: 機能・セキュリティ・運用準備性を自動検証

開発者は各ステップでAI生成物を検証し、重要判断を行う。ブラウンフィールドでは、既存コードをまずセマンティックに豊かなモデル表現（静的モデル/動的モデル）へ昇格させ、AI文脈を簡潔かつ正確化する。AI-DLCでは、このフェーズも可能なら同室のMob Constructionとして進めることを推奨する。

#### Operations Phase

デプロイ、可観測性、保守をAIで効率化する。

- AIがメトリクス・ログ・トレースを分析し、異常やSLA違反予兆を検知
- 事前定義ランブックと連携し、スケーリング、チューニング、障害分離等を提案
- 開発者承認後、AIが解決アクションを実行可能
- 開発者はSLA/コンプライアンス適合を検証する

### 3. ワークフロー（Workflow）

ビジネス意図（例: グリーンフィールド開発、ブラウンフィールド拡張、モダナイゼーション、欠陥修正）を入力すると、AIがまずLevel 1 Plan（実装ワークフロー）を提案する。人間が透明にレビュー・検証・修正し、事業目標と工学制約に合わせる。

AI-DLCの中核は、各ステップ成果物を次ステップ向けのセマンティックに豊かな文脈へ段階的に磨き上げることにある。各ステップは戦略的判断点であり、人間の監督は損失関数のように早期誤りを補正する。この分解と検証は再帰的に繰り返される。

生成された成果物（Intent、ユーザーストーリー、ドメインモデル、テスト計画など）は永続化され、ライフサイクル全体でAIの「文脈記憶」として参照される。成果物同士はリンクされ、前後トレーサビリティ（例: ドメインモデル要素と特定ストーリーの接続）を確保し、常に適切文脈を取得できる。

## IV. AI-DLC実践例: グリーンフィールド開発

例として、POが「クロスセル向け推薦エンジンを開発する」という高レベル意図を提示するケースを扱う。AIはこれを新規アプリ構築意図として認識し、Level 1 Planを生成。チームが検証・修正後、Inceptionに進む。

### 1. Inception Phase（Mob Elaboration）

- AIが明確化質問を行う（例: 主要利用者は誰か、主要成果は何か）
- AIが意図をユーザーストーリー、NFR、リスク記述へ展開
- チームが検証し修正を反映
- AIが高凝集なストーリーをUnitへ編成
  - 例: ユーザーデータ収集、推薦アルゴリズム選定、API統合
- POが出力を検証・調整
  - 例: データ収集UnitにGDPR対応詳細を追加
- AIが必要に応じPRFAQを生成
- 開発者/POがPRFAQとリスクを検証

### 2. Construction Phase（Mob Programming/Mob Testing）

- 開発者がAIセッション開始、AIが担当Unit着手を促す
- AIがDDDで業務ロジックをモデル化
  - 例: Product、Customer、Purchase Historyの関係定義
- 開発者が現実シナリオと整合するようモデルを検証
- AIが論理設計へ変換し、スケーラビリティ/耐障害性を適用
  - 例: イベント駆動設計、AWS Lambda提案
- 開発者がトレードオフを評価し必要に応じ上書き
  - 例: 計算はLambda採用、保存はクエリ性能重視でDynamoDB選択
- AIが各Unitの実行コードを生成（AWSサービスへ対応付け）
- AIが機能・セキュリティ・性能テストを自動生成
  - 例: 協調フィルタリング実装とDynamoDB連携
- 開発者が生成コードとテストケースをレビュー・調整

テストと検証:

- AIが全テストを実行・分析し課題を可視化
- 失敗に対する修正案を提示（例: クエリ最適化）
- 開発者が妥当性確認し、必要に応じ再実行

### 3. Operations Phase

デプロイ:

- AIがDeployment Unitへパッケージ化
- 開発者が設定承認し、ステージング/本番へ展開

可観測性と監視:

- AIがメトリクス・ログ・トレースを分析し、異常とSLA予兆を検知
  - 例: ピーク時遅延増大を検知しスケール提案
- AIが運用プレイブックと連携して対処案を提示
  - 例: DynamoDBスループット増強、API Gatewayトラフィック再配分
- 開発者が提案を承認し、解決結果を監視

ブラウンフィールドとは、既存システムに対する新機能追加、NFR最適化、技術負債対処（リファクタ/欠陥修正）を指す。

### 1. Inception Phase

活動はグリーンフィールドと同様。

### 2. Construction Phase

- AIが既存コードを高レベルモデルへ昇格
  - 静的モデル: コンポーネント、責務、関係
  - 動的モデル: 主要ユースケースを実現する相互作用
- 開発者とPMが逆解析モデルをレビュー・検証・補正
- その後の工程はグリーンフィールドと同様

### 3. Operations Phase

活動はグリーンフィールドと同様。

## VI. AI-DLCの導入（Adopting AI-DLC）

AI-DLCは既存Agileから大きく逸脱せず、導入容易性を重視して設計されている。とはいえ、従来手法を長年運用してきた組織や独自AIネイティブ手法を模索中の組織には導入戦略が必要である。以下の2アプローチが有効とされる。

- 実践しながら学ぶ（Learning by Practicing）
  - AI-DLCはMob ElaborationやMob Construction等の儀式群であり、文書学習より実地演習が有効
  - 実案件シナリオでガイド付き実践することで定着を加速
  - AWS Solution Architectsは大規模導入向けにAI-DLC Unicorn Gymを提供
- 開発者体験ツールへ埋め込む
  - 顧客はSDLC横断オーケストレーションツールを構築中（例: FlowSource、CodeSpell、AIForce）
  - これらへAI-DLCを埋め込むと、大規模組織でも大規模な導入施策なしで自然に実践可能

## 付録A（Appendix A）: AI-DLC実践のためのプロンプト集（日本語訳）

以下は原文に掲載された「AIと対話してAI-DLCを実践するためのプロンプト」の翻訳です。

### Setup Prompt

今日はアプリケーションを構築します。フロントエンドおよびバックエンドの各コンポーネントごとにプロジェクトフォルダを作成してください。すべての文書はaidlc-docsフォルダに置きます。セッション中、私はあなたに事前計画を依頼し、計画をmdファイルで作成してもらいます。あなたは私が計画を承認した後にのみ作業できます。計画は常にaidlc-docs/plansフォルダに保存します。

作成する文書はすべてmd形式です。要件や機能変更文書はaidlc-docs/requirementsフォルダ、ユーザーストーリーはaidlc-docs/story-artifactsフォルダ、アーキテクチャ/設計文書はaidlc-docs/design-artifactsフォルダに保存します。すべてのプロンプト（順序付き）はaidlc-docs/prompts.mdに保存します。

このプロンプトの理解を確認し、保存に必要なフォルダとファイルが未作成なら作成してください。

### Inception / User stories

あなたの役割: あなたは熟練のプロダクトマネージャーであり、以下Taskのシステム開発契約となる明確なユーザーストーリーを作成します。まず作業計画を立て、user_stories_plan.mdに各ステップをチェックボックス付きで記載してください。確認が必要なステップには、私への確認事項を注記してください。重要な決定を独断で行ってはいけません。計画完成後、私のレビューと承認を求めてください。承認後、同じ計画を1ステップずつ実行し、完了時にチェックを付けてください。

あなたのTask: ここに記載された高レベル要件に基づいてユーザーストーリーを作成する（ここにプロダクト概要を記載）

レビューと必要な修正の後の返答例:
