# D: 従来手法との比較

AI-DLC を「これまでの開発と何が違うのか」という視点で理解するための比較資料。

---

## 1. 一言で言うと何が違うか

```
ウォーターフォール: 「人間が設計して人間が書く」
Scrum:             「人間が設計して人間が書く（短い反復で）」
AI-DLC:            「人間が意図を定義して AI が生成し人間が判断する」
                                              ↑ここが根本的に違う
```

---

## 2. フェーズ比較

### ウォーターフォール vs AI-DLC

| フェーズ | ウォーターフォール | AI-DLC |
|---------|-----------------|--------|
| 要件定義 | ドキュメントを人間が書く（週〜月単位） | Mob Elaboration で AI と協働（時間単位） |
| 設計 | アーキテクトが単独で設計（数週間） | Domain Design → Logical Design を Mob で（日単位） |
| 実装 | 各開発者が個別に担当 | Mob Construction で AI がコード生成・人間がレビュー |
| テスト | 実装後にテストフェーズが始まる | Bolt ごとに即テスト（実装と並行） |
| リリース | 大規模まとめリリース（月〜年単位） | Unit 単位の段階リリース（週単位） |

### Scrum vs AI-DLC

| 比較軸 | Scrum | AI-DLC |
|--------|-------|--------|
| 反復単位 | スプリント（1〜4週間） | Bolt（時間〜日）→ Unit（週）→ Intent（月） |
| 要件定義者 | PO が User Story を書く | PO + チーム全員 + AI が Mob で協働 |
| 実装担当 | 開発者が個人で担当 | AI が生成 → チームが Mob でレビュー |
| コードレビュー | PR ベースで非同期 | Mob Construction で同期・リアルタイム |
| ドキュメント | 最小限（Agile Manifesto） | Intent / Unit 仕様は明示的に必須 |
| AI の役割 | ツールとして個人が使う | 開発プロセスに統合（オーケストレーター） |

---

## 3. 役割の変化

### 従来のソフトウェアエンジニア
```
書く → レビューされる → 修正する → マージ
  ↑
個人の実装力が生産性に直結
```

### AI-DLC でのソフトウェアエンジニア
```
Intent を定義 → AI に指示 → 生成物をレビュー → 判断・修正指示
  ↑                              ↑
「何を作るか」の判断力           「良いコードかの判断力」
```

**変化まとめ**:

| 役割 | 従来 | AI-DLC |
|------|------|--------|
| 開発者 | コードを書く人 | AI の出力を判断・方向付けする人 |
| PO | 要件を書く人 | Intent の明確化をリードする人 |
| テックリード | 設計の権威者 | Navigator として AI と Mob の調整役 |
| QA | テストを書く人 | テスト戦略を定義し AI にテストを書かせる人 |
| アーキテクト | 技術選定の最終判断者 | ADR で意思決定の根拠を記録する人（変わらず重要）|

## 4. 生産性・品質の違い

### 変更リードタイム（コミットから本番まで）

```
ウォーターフォール: ━━━━━━━━━━━━━━━━━━━━━━━━  数ヶ月〜年
Scrum (標準):       ━━━━━━━━━━━━             数週間
AI-DLC (目標):      ━━━━                     数時間〜数日
```

### コード品質への影響

| 品質軸 | 従来 Scrum | AI-DLC |
|--------|-----------|--------|
| コードの一貫性 | 開発者によりバラつく | AI が生成するため高い均一性 |
| テストカバレッジ | 80%達成が一般的 | AI がテストも生成するため高まりやすい |
| セキュリティ | レビューで発見 | プロンプトに OWASP を組み込んで生成時に対処 |
| ドキュメント | 後追いになりがち | Bolt 単位でドキュメントも AI が生成 |

---

## 5. 「AI-DLC は Scrum を置き換えるのか？」という疑問

**答え: いいえ。AI-DLC は Scrum の上に乗る概念**

```
                    ┌─────────────────────┐
                    │       AI-DLC        │ ← AI との協働方法論
                    │  Intent/Unit/Bolt   │
                    └──────────┬──────────┘
                               │ 組み合わせて使う
                    ┌──────────▼──────────┐
                    │        Scrum        │ ← チームの動き方の方法論
                    │ Sprint/Backlog/PO   │
                    └─────────────────────┘
```

**実際の組み合わせ例**:
- Sprint = 2 週間、中身は Unit を構成する Bolt を消化
- Sprint Backlog = Bolt リスト
- Sprint Review = Unit の受け入れ基準を確認
- Retrospective = AI プロンプトの質・Mob の進め方を振り返る

---

## 6. AI-DLC を導入するべきケース / 向かないケース

### 導入が効果的なケース

- 新機能開発が多い（グリーンフィールド・新規機能）
- 要件の曖昧さが多く、早期の方向合わせが必要
- チームに Mob Programming の文化・習慣がある（または作りたい）
- AI ツールを個人でバラバラに使っており、チームとして統一したい

### 注意が必要なケース

- 厳密な規制要件（コード変更の全履歴管理など）がある → ADR と監査ログを強化して対処可能
- チームが分散・非同期作業が前提 → Mob は難しいが非同期 Mob のアレンジが必要

### 向かないケース

- チームメンバーが AI の出力を批判的にレビューできない → まず AI リテラシーを先に高める
- 「AI に全部任せる」思想のチーム → AI-DLC は AI を制御する方法論。任せきりは目指していない

---

## 7. 一行まとめ

| 手法 | 一行まとめ |
|------|-----------|
| ウォーターフォール | 計画を完璧にしてから作る |
| Scrum | 短い反復で作りながら計画を修正する |
| **AI-DLC** | **意図を定義し AI に作らせ、人間が判断する** |
