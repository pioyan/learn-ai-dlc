# F: よくある誤解とアンチパターン集（学習者向け）

AI-DLC を学び始めた人が陥りやすい**概念の誤解**と**実践上のアンチパターン**を整理。  
「そういうことか！」という気づきに使ってください。

---

## 第1章：概念の誤解

### 誤解 1: 「AI-DLC = AI にコードを書かせる手法」

**よくある誤解**:
> 「AI に全部書かせて、人間は確認するだけ？」

**実際**:
AI-DLC の核心は「**意図の定義**」と「**判断**」。  
コード生成は手段のひとつに過ぎない。

```
誤解した場合のやり方:
  AI に「アプリを作って」→ コードが出てくる → そのまま使う → 品質問題が頻発

正しいやり方:
  Intent を明確に定義 → Unit に分解 → Bolt に細分化
  → AI に Bolt 単位でコード生成させる → チームでレビュー・判断 → 次の Bolt へ
```

**学習のポイント**: Intent → Unit → Bolt の分解こそが AI-DLC の価値の源泉

---

### 誤解 2: 「Mob = 全員が集まってコードを書くだけ」

**よくある誤解**:
> 「モブプログラミングって、みんなで 1 台のパソコンを見てるやつでしょ？」

**実際**:
AI-DLC の Mob は「**AI を含む知識統合の場**」。  
コードを書くだけでなく、Design → Decision → Review → Test をすべてMobで行う。

```
Mob Elaboration: 要件・設計の意思決定を Mob で
Mob Construction: 実装・テスト・レビューを Mob で
```

**何が違うか**:
- 個人作業と違い「なぜそう設計したか」が全員に見える
- AI への指示も Mob で合意 → 「AI の暴走」を防げる

---

### 誤解 3: 「Intent は最初に決まったら変えない」

**よくある誤解**:
> 「Intent = 要件定義書。最初に決めたら動かせない」

**実際**:
Intent は **生きたドキュメント**。学びが増えるにつれて更新する。

```
典型的な Intent の進化:

Week 1 Intent:
  「推薦エンジンで購買率を向上させる」

Week 3（実装してわかった後）の Intent:
  「クロスセルに特化した推薦エンジンで、
   購買単価を 15% 向上させる（完全なパーソナライズは次フェーズ）」
```

**学習のポイント**: Intent が変わったら Unit/Bolt も見直す。それは失敗ではなく学習

---

### 誤解 4: 「Unit = マイクロサービス」

**よくある誤解**:
> 「Unit ごとに別のサービスとして独立デプロイするんでしょ？」

**実際**:
Unit は **論理的な分割単位**。物理的なサービス境界とは別の概念。

```
同一モノリス内でも Unit は複数持てる:
  Intent: 推薦エンジン
    Unit-A: メール通知（同一サービス内の別モジュール）
    Unit-B: 行動ログ（同一サービス内の別モジュール）
    Unit-C: アルゴリズムコア（別 Lambda として独立）

Unit-A/B はモノリス内、Unit-C のみ独立サービス → すべて Unit
```

---

### 誤解 5: 「AI が生成したコードはそのまま使ってよい」

**よくある誤解**:
> 「AI が生成してくれたんだから、きっと正しい」

**実際**:
AI は **確率的な出力**をする。正しそうに見えて間違っていることが多い。

```
AI がよく犯すミス:
- 架空の API・ライブラリを存在するかのように書く（Hallucination）
- セキュリティのベストプラクティスを無視したコードを出す
- ドメイン固有のルールを知らないまま汎用コードを出す
- 最新の AWS サービスの仕様変更を知らないまま古い書き方をする
```

**学習のポイント**: AI の出力は「ドラフト（草案）」。必ずチームでレビューする

## 第2章：実践上のアンチパターン

### アンチパターン 1: Intent がふわっとしたまま進む（Vague Intent）

**症状**:
```
Intent: 「ユーザー体験を改善する」
      ↓ 曖昧すぎる
Unit 分解が「全部に関係する」になり分解できない
```

**対策**:
Intent に必ず以下を含める:
- 定量的な成功指標（KPI）
- 対象ユーザーの具体像
- 「スコープ外」を明示する（何をしないかが重要）

---

### アンチパターン 2: Mob がコードレビュー会議になる

**症状**:
```
Mob Construction のつもりが...
「このコード、ちょっと読んでおいて」
→ 各自がコードを読む時間になる
→ 意見がバラバラに出る
→ 議論が迷子になる
```

**対策**:
- Mob Construction はリアルタイムで AI に指示しながら進める
- 事前にコードを読まない（Mob の場で初めて見る）
- Navigator が方向性を維持する責任を持つ

---

### アンチパターン 3: Bolt が大きすぎる（Mega Bolt）

**症状**:
```
Bolt-1: 「推薦エンジン全体を実装する」（3週間）
→ 進捗が見えない
→ レビューが大変
→ AI への指示が曖昧になる
```

**適切な Bolt のサイズ**:
```
Bolt-C1: DynamoDB スキーマ設計（4時間）
Bolt-C2: データ投入ロジック（4時間）
Bolt-C3: 協調フィルタリング関数（8時間）
Bolt-C4: キャッシュ層（4時間）
Bolt-C5: Lambda ハンドラー（4時間）
Bolt-C6: API テスト（4時間）
```

**目安**: 1 Bolt = 最大 8 時間（1 営業日）

---

### アンチパターン 4: AI プロンプトを毎回ゼロから書く

**症状**:
```
毎回: 「えっと、なんて聞けばいいんだっけ...」
→ プロンプト作成に時間がかかる
→ チームごとにバラバラな品質
```

**対策**:
[E_プロンプト設計基礎.md] のテンプレートを使い回す。  
チームで「このプロンプトが良かった」を共有・蓄積する仕組みを作る

---

### アンチパターン 5: レビューをスキップする（Trust the AI）

**症状**:
```
「AI が生成してくれたし、動いたから OK」
→ コードレビューをスキップ
→ セキュリティの脆弱性が残る
→ ドメインロジックの間違いが残る
→ 本番でインシデント発生
```

**対策**:
Mob Construction のレビューチェックリストを必ず実施:
- [ ] 業務ロジックが仕様通りか
- [ ] エラーハンドリングがあるか
- [ ] セキュリティ（認証・認可・入力値検証）は適切か
- [ ] テストが通っているか
- [ ] 意図しない副作用はないか

---

### アンチパターン 6: Mob を「誰でも参加できる会議」にする

**症状**:
```
10人が参加
→ 全員が意見を言う
→ 議論が収束しない
→ 3時間経っても何も決まらない
```

**適切な Mob の人数と役割**:
```
最適: 4〜6 名
  - Navigator: 1名（テックリード）
  - Driver: 1名（ローテーション）
  - Mob: 2〜4名

7名以上: 2つの Mob に分けるか、一部はオブザーバーとして参加
```

---

### アンチパターン 7: ADR を書かない（暗黙の意思決定）

**症状**:
```
「なんでここ DynamoDB 使ってるんだっけ？」
「誰かが決めた気がするけど理由が分からない」
→ 後から設計変更が難しくなる
→ 新メンバーが理解できない
```

**対策**:
重要な技術選定は必ず ADR（Architecture Decision Record）を残す:
```markdown
# ADR-001: DynamoDB を推薦エンジンのデータストアに採用

## 状況
推薦エンジンのデータストアを選定する

## 決定
Amazon DynamoDB を採用する

## 理由
- 読み取り中心のワークロード（読み:書き = 9:1）
- スケーリングが自動（10,000 TPS に対応が必要）
- 推薦結果の TTL 管理が容易

## トレードオフ
- 複雑なクエリが難しい（JOIN 不可）
- コスト予測が難しい場合がある

## 結果
Unit-C で採用。パフォーマンステストでは p99 = 8ms を達成
```

## チェックリスト：学習時の確認

自分が陥っていないかを確認:

- [ ] Intent に定量的 KPI が含まれている
- [ ] Bolt が 8 時間以内に収まっている
- [ ] AI の出力を必ずレビューしている
- [ ] Mob の人数が 6 名以内に抑えられている
- [ ] 重要な技術選定に ADR がある
- [ ] Intent/Unit 定義が最新の状態に更新されている
