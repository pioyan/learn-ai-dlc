# B: AI-DLC 用語集

AI-DLC のドキュメントに登場する専門用語の定義。アルファベット順・日本語順で引けます。

---

## AI-DLC コア概念

### Intent（インテント）
**定義**: 開発プロジェクトの「目的・背景・成功基準」をまとめた最上位のドキュメント  
**別名**: Intent ドキュメント、Why ドキュメント  
**含む内容**: ビジネス目標、対象ユーザー、成功指標（KPI）、制約条件、スコープ外  
**類似概念**: プロジェクト憲章（従来）、PRFAQ（Amazon）
**補足**: 本リポジトリでは Intent を「概念」ではなく、実際に残して更新する文書として扱う

**例**:
```
Intent: ECサイトの推薦エンジン
- 目標: 顧客の購買単価を 15% 向上
- 対象ユーザー: 既存会員（月間アクティブ100万人）
- 成功指標: クリック率 3%以上、購買転換率 1.5%以上
- 制約: 既存 API に破壊的変更なし
```

---

### Unit（ユニット）
**定義**: Intent を実現するための「独立して構築・テスト・デプロイできる機能単位」  
**サイズ感**: 1〜4 週間で実装できる範囲（Bolt の集合体）  
**分割基準**: 独立性（他 Unit に依存しない）・責任の明確さ・チーム単位で担当できる粒度

**例**:
```
Intent: 推薦エンジン
  ├─ Unit-A: メール通知サービス（Week 1-2）
  ├─ Unit-B: ユーザー行動ログ収集（Week 1-3）
  └─ Unit-C: 推薦アルゴリズムコア（Week 2-4）
```

---

### Bolt（ボルト）
**定義**: Unit をさらに細分化した「数時間〜1日で完了できる最小実装単位」  
**サイズ感**: 2〜8 時間  
**目的**: 進捗の可視化、AI との協調作業の単位

**例**:
```
Unit-C: 推薦アルゴリズムコア
  ├─ Bolt-C1: DynamoDB スキーマ設計と作成（4h）
  ├─ Bolt-C2: 協調フィルタリング実装（8h）
  ├─ Bolt-C3: ElastiCache キャッシュ層（4h）
  └─ Bolt-C4: Lambda ハンドラー & API Gateway（4h）
```

---

### Mob Elaboration（モブ エラボレーション）
**定義**: チーム全員で Intent の明確化・User Story 生成・Unit/Bolt 分解を行うワークショップ  
**別名**: Inception フェーズ、要件定義 Mob  
**所要時間**: 4〜8 時間（通常 1 日）  
**参加者**: PO、テックリード、開発者、QA、ドメイン専門家（5〜8名）

---

### Mob Construction（モブ コンストラクション）
**定義**: AI を活用しながらチームで実装・テストを進める開発セッション  
**別名**: Construction フェーズ、実装 Mob  
**サイクル**: Domain Design → Logical Design → Code Generation → Review → Test  
**参加者**: テックリード（Navigator）、開発者 2〜4 名（Driver ローテーション）

---

### Driver / Navigator（ドライバー / ナビゲーター）
**定義**: Mob Programming の役割分担

| 役割 | 責任 | 交代頻度 |
|------|------|---------|
| **Driver** | キーボードを操作する人。AIへの指示入力も担当 | 15〜25 分ごとに交代 |
| **Navigator** | 全体設計・方向性を指示する人。最も経験豊富な人が担当 | 1 セッション固定が多い |
| **Mob** | 残りの参加者。意見・提案・レビューを行う | — |

---

### PRFAQ（プレスリリース + FAQ）
**定義**: Amazon が発明した「完成した製品のプレスリリース」を先に書いて方向性を決める手法  
**AI-DLC での使い方**: Intent ドキュメントを書く際のフォーマットとして活用  
**目的**: 完成後の姿を先に具体化し、Intent の解像度を上げる  
**構成**:
- PR（プレスリリース）: 完成時に発表される想定の記事
- FAQ（よくある質問）: 内部（開発チーム向け）と外部（顧客向け）

---

### NFR（Non-Functional Requirements）
**定義**: 非機能要件。「何ができるか」ではなく「どの品質水準で動くか」を定義

| 分類 | 例 |
|------|---|
| Performance | レスポンス時間 p99 < 200ms |
| Availability | SLA 99.9%（月間ダウン < 44分） |
| Security | 認証: OAuth 2.0 + JWT 必須 |
| Scalability | 同時接続 10,000 ユーザー |
| Observability | 全トランザクションのログ・トレース |

---

### Risk Register（リスクレジスター）
**定義**: プロジェクトで想定されるリスクを一覧化し、対策を事前に定義したドキュメント  
**記録内容**: リスク名・確率・影響度・対策・オーナー

---

### ADR（Architecture Decision Record）
**定義**: アーキテクチャ上の重要な意思決定を記録するドキュメント  
**目的**: 「なぜその技術を選んだか」を後から分かるようにする  
**フォーマット**: タイトル / 状況 / 決定内容 / 理由 / トレードオフ / 結果

---

## 開発アプローチ・設計用語

### Greenfield（グリーンフィールド）
**定義**: 既存システムの制約がほぼない状態で進める新規開発  
**対義語**: Brownfield

### Brownfield（ブラウンフィールド）
**定義**: 既存システムや既存運用を前提に、機能追加・改修・移行を行う開発  
**ポイント**: 互換性、既存データ、段階移行の制約を考慮する

### Context Engineering
**定義**: AI に必要な文脈を、必要なタイミングで、必要な粒度で渡す設計  
**例**: Intent → Unit → Design → Code の成果物を段階的に渡す

### 設計技法
**定義**: ドメインや要件を整理し、良い構造へ導くための方法論や進め方  
**例**: DDD, TDD, BDD

### 設計パターン
**定義**: 繰り返し現れる設計上の問題に対する再利用可能な解決策  
**例**: CQRS, Circuit Breaker  
**違い**: 設計技法が進め方の枠組みなのに対し、設計パターンは個別問題への解決策

### ユースケース
**定義**: ユーザーや外部システムが、ある目的のためにシステムとどう関わるかを表した利用場面

### シナリオ
**定義**: ユースケースの具体的な実行例や文脈  
**違い**: ユースケースが抽象的な利用目的、シナリオが具体的な進行例

### スコープ
**定義**: プロジェクトや Unit で「何をやるか / 何をやらないか」の範囲

### Event Storming
**定義**: 業務イベントを起点に、コマンド、集約、サービス境界を整理するワークショップ手法

### Service Boundary（サービス境界）
**定義**: どこまでを1つのサービスの責任範囲とするかを決める境界  
**判断材料**: 変更理由、データ所有権、障害分離、チーム責任

### Static Model（静的モデル）
**定義**: コンポーネントやエンティティ、責務、関係など、構造を表すモデル

### Dynamic Model（動的モデル）
**定義**: ユースケース実行時の相互作用や処理順序など、振る舞いを表すモデル

### Policy（イベントポリシー）
**定義**: あるイベントを受けて、次のコマンドや処理を起動するルール  
**例**: OrderCreatedEvent を受けたら InitiatePayment を発行する

### Canary Deploy（カナリアデプロイ）
**定義**: 一部ユーザーや一部トラフィックだけ新バージョンに切り替え、問題がなければ段階的に拡大するリリース方法

### Big Bang Release（ビッグバンリリース）
**定義**: 旧システムから新システムへ一括で切り替えるリリース方法  
**対比**: 段階移行や Canary Deploy よりも切り替えは速いが、失敗時の影響が大きい

### Kafka
**定義**: 大量イベントを高スループットで処理するメッセージング / ストリーミング基盤  
**補足**: EventBridge がマネージドなイベント配信に向くのに対し、Kafka は細かな制御や高スループット処理に向く

## DDD（Domain-Driven Design）用語

### Bounded Context（境界付けられたコンテキスト）
**定義**: 特定のドメイン概念（言語・モデル）が一貫して通用する範囲  
**例**: 「Order Context」では Order = 注文。「Shipping Context」では Order = 出荷指示

### Aggregate（集約）
**定義**: 一緒に変更される Entity のまとまり。一貫性の境界  
**例**: `Order` Aggregate = Order + OrderItems（OrderItem 単体では保存しない）

### Entity（エンティティ）
**定義**: 一意の識別子（ID）を持ち、時間とともに変化するオブジェクト  
**例**: `Customer(customer_id=123)` — 名前が変わっても同一顧客

### Value Object（値オブジェクト）
**定義**: 識別子を持たず、値そのものが意味を持つオブジェクト。不変  
**例**: `Money(amount=1000, currency="JPY")` — 同じ値なら同一とみなす

### Domain Event（ドメインイベント）
**定義**: ドメイン内で起きた「重要な出来事」を表すオブジェクト  
**命名規則**: 過去形（OrderCreated, PaymentProcessed）  
**目的**: サービス間の疎結合な連携

### Repository（リポジトリ）
**定義**: Aggregate の永続化・取得を担うインターフェース（データストアを隠蔽）

### Ubiquitous Language（ユビキタス言語）
**定義**: ドメインの専門家と開発者が共通で使う言葉。コードにもそのまま使う

## 実装・レガシー用語

### DTO（Data Transfer Object）
**定義**: 画面や API に渡すためのデータ転送用オブジェクト。ドメインモデルと分けて使うことが多い

### DAO（Data Access Object）
**定義**: データベースアクセス処理をまとめたオブジェクトや層。Repository パターンよりストレージ寄りの責務を持つことが多い

### ORM（Object-Relational Mapping）
**定義**: オブジェクトとリレーショナルデータベースの対応付けを自動化する仕組み  
**例**: Hibernate, JPA, SQLAlchemy

### CSR（Customer Service Representative）
**定義**: 顧客対応を行う担当者。コールセンターやサポート窓口のオペレーターを指す

### API（Application Programming Interface）
**定義**: ソフトウェア同士がやり取りするためのインターフェース  
**補足**: 本リポジトリでは REST API を指すことが多い

### ステージング環境
**定義**: 本番に近い構成で、リリース前検証を行う環境

### テスト種別
**定義**: 品質確認の目的ごとに分けたテストの分類  
**代表例**: Unit Test、Integration Test、E2E Test、負荷テスト

### エラーハンドリング
**定義**: エラーをどう扱い、利用者・ログ・再試行にどう反映するかという方針全体

### 例外処理
**定義**: try/catch などで実際に例外を捕捉・変換するプログラム上の処理

## メトリクス用語（DORA）

### KPI（Key Performance Indicator）
**定義**: 成功指標のうち、継続的に追跡する主要指標  
**補足**: 「成功指標」は目的達成を測る全体概念で、KPI はその中でも日常的に監視する中心指標

### DORA 4 Keys（DORA 4指標）
DevOps Research and Assessment が定義したソフトウェア開発の4指標：

| 指標 | 意味 | 優秀な水準 |
|------|------|-----------|
| **変更リードタイム** | コミット → 本番リリースまでの時間 | 1時間以内 |
| **デプロイ頻度** | 本番へのリリース頻度 | オンデマンド（複数回/日） |
| **変更失敗率** | デプロイがインシデントを引き起こす割合 | 5% 未満 |
| **MTTR** | インシデント検知 → 復旧までの時間 | 1時間以内 |

### p99（99パーセンタイル）
**定義**: 100回のリクエストのうち99回がこの値以下で終わることを示す指標  
**使い方**: 平均値よりも「遅い少数のリクエスト」を把握しやすい

### SLA（Service Level Agreement）
**定義**: サービス提供側が可用性や応答時間などについて利用者に約束する水準  
**例**: 可用性 99.9%

## セキュリティ用語

### Threat Model（脅威モデル）
**定義**: システムに対して起こりうる脅威、影響、対策を整理したモデル  
**関係**: Risk Register より具体的で、STRIDE は Threat Model を作るときの観点として使われる

### STRIDE
脅威モデリングのフレームワーク（Spoofing / Tampering / Repudiation / Information Disclosure / Denial of Service / Elevation of Privilege）

### ログ（Log）
**定義**: 実行時に発生した出来事を記録する一般的な記録  
**違い**: 監査ログは証跡用途、トレースは処理経路の可視化に使う

### 認証（Authentication）
**定義**: アクセスしてきた相手が誰かを確認すること

### 認可（Authorization）
**定義**: 確認された相手に何を許可するかを決めること

### RBAC（Role-Based Access Control）
**定義**: 役割単位で権限を割り当てるアクセス制御方式

### 監査ログ（Audit Log）
**定義**: 誰が、いつ、何をしたかを追跡できるように残すログ。法務・監査証跡向け

### トレース（Trace / Distributed Tracing）
**定義**: 1つのリクエストが複数サービスをどう通過したかを追跡する記録  
**違い**: ログは個別事象、監査ログは証跡、トレースは処理経路の可視化

### mTLS（Mutual TLS）
サーバーとクライアントが互いに証明書を検証する双方向 TLS。サービス間通信の認証に使う

### Zero Trust Architecture
「すべてを信頼しない」前提で設計するセキュリティアーキテクチャ。内部ネットワークも常に検証

### PCI DSS
クレジットカード情報を扱う事業者が準拠すべきセキュリティ基準

## AWS サービス用語（AI-DLC での役割）

| サービス | 役割 |
|---------|------|
| **Lambda** | サーバーレス関数実行（Bolt の実装単位） |
| **DynamoDB** | NoSQL データベース（高スループット・スケーラブル） |
| **EventBridge** | イベント駆動ルーティング（マイクロサービス連携） |
| **ElastiCache** | Redis/Memcached キャッシュ（レイテンシ削減） |
| **X-Ray** | 分散トレーシング（パフォーマンス可視化） |
| **CloudWatch** | ログ・メトリクス・アラート監視 |
| **KMS** | 暗号化キー管理（鍵のローテーション自動化） |
| **Secrets Manager** | DB パスワード等の機密情報管理 |
