## フェーズ1クイズ（基礎理解）

### Q1: 概念の定義

AI-DLC の 3 つの主要フェーズを順番に答えてください。

<details>
<summary>▶ 答えを見る</summary>

**Inception（インセプション）→ Construction（コンストラクション）→ Operations（オペレーションズ）**

- **Inception**: Intent 定義・Unit/Bolt 分解を行う Mob Elaboration フェーズ
- **Construction**: 実際に実装・テストを行う Mob Construction フェーズ
- **Operations**: 本番稼働後の監視・改善フェーズ

</details>

### Q2: 用語の区別

以下の 3 つの違いを説明してください。

- **Intent**（インテント）
- **Unit**（ユニット）
- **Bolt**（ボルト）

<details>
<summary>▶ 答えを見る</summary>

**Intent**: プロジェクト全体の「なぜ・何を・どのレベルで」を定義する最上位ドキュメント。ビジネス目標・成功指標を含む。（例: 「推薦エンジンで購買単価を 15% 向上させる」）

**Unit**: Intent を実現するための独立した機能単位。1〜4 週間で構築できる粒度。他の Unit に依存しない。（例: 「推薦アルゴリズムコア」）

**Bolt**: Unit をさらに細分化した最小実装単位。2〜8 時間で完了できる粒度。AI との協働の単位。（例: 「DynamoDB スキーマ設計」）

**関係**: Intent（月）→ Unit（週）→ Bolt（時間〜日）という階層構造

</details>

### Q3: 役割の理解

Mob Construction における「Navigator」と「Driver」の役割の違いを説明してください。

<details>
<summary>▶ 答えを見る</summary>

**Driver**: キーボードを操作する人。AI へのプロンプト入力も担当。15〜25 分ごとにローテーションする。

**Navigator**: 全体の設計・方向性を指示する人。最も経験豊富なメンバーが担当することが多い。「何を作るべきか」を決める。

**ポイント**: Driver は「どうやって書くか」を考え、Navigator は「何を作るべきか」を考える。この分離が重要。

</details>

### Q4: 比較問題

AI-DLC と Scrum の「最大の違い」は何か？ 1 文で答えてください。

<details>
<summary>▶ 答えを見る</summary>

**「Scrum は人間が設計・実装するプロセスを管理するのに対し、AI-DLC は AI が生成し人間が判断するという、AI 協働を前提とした開発方法論である」**

</details>

### Q5: 誤解を見抜く

以下のチームの発言の中で、**AI-DLC の誤解が含まれているもの**を選び、何が間違っているかを説明してください。

> A: 「まず Intent ドキュメントに KPI を明記しましょう」  
> B: 「AI が生成したコードはレビュー不要です。AI が正しいはずなので」  
> C: 「Bolt のサイズは 8 時間以内にしましょう」  
> D: 「Intent は学びに応じて更新してよいです」  

<details>
<summary>▶ 答えを見る</summary>

**B が誤解**

AI の出力は確率的なものであり、正確であることを保証しない。ハルシネーション（架空の情報）、セキュリティの見落とし、ドメイン固有ルールの無視などが起こりうる。

AI-DLC では「AI の出力は必ずチームでレビューする」ことが前提。

</details>

## フェーズ2クイズ（実践理解）

### Q6: プロンプト設計

以下の 2 つのプロンプトを比較し、どちらが良いプロンプトか、その理由も含めて答えてください。

**プロンプト X**:
```
「ユーザー認証を実装してください」
```

**プロンプト Y**:
```
【Context】
ECサイトのバックエンド。Python 3.11 + FastAPI + JWT 認証。
既存ユーザーテーブルは PostgreSQL の users テーブル（id, email, password_hash）。

【Role】
あなたはシニア Python エンジニアです。

【Task】
ユーザーログインエンドポイントを実装してください。
メール + パスワード認証、成功時に JWT トークンを返す。

【Constraints】
- パスワードは bcrypt でハッシュ検証すること
- JWT の有効期限は 24 時間
- 失敗時は具体的なエラー理由を返さない（セキュリティ）
- pytest でテストも含める
```

<details>
<summary>▶ 答えを見る</summary>

**プロンプト Y が優れている**

理由:
1. **Context**: 技術スタック・既存テーブル構造が明示されており、AI が一貫した実装を生成できる
2. **Role**: 専門家として回答させることで品質が上がる
3. **Task**: 何を実装するかが具体的（ログイン → JWT 返却）
4. **Constraints**: セキュリティ要件（bcrypt・JWT 有効期限・エラーメッセージ）が明示されており、安全な実装になる

プロンプト X の問題: 何の言語か・どのフレームワークか・セキュリティ要件は何かが不明で、AI が「それっぽいが間違った」コードを出す可能性が高い

</details>

### Q7: Unit 分解の判断

以下の機能リストを「適切な Unit 分割案」に整理してください。

機能リスト:
- ユーザー登録フォーム
- ユーザー認証（ログイン/ログアウト）
- ユーザープロフィール編集
- 商品一覧ページ
- 商品詳細ページ
- カートに追加
- 注文確定
- 支払い処理
- 注文履歴
- メール通知（注文確認・発送）

<details>
<summary>▶ 答えを見る</summary>

**分割例（正解は複数ある）**:

```
Unit-A: ユーザー管理
  - ユーザー登録フォーム
  - ユーザー認証（ログイン/ログアウト）
  - ユーザープロフィール編集

Unit-B: 商品カタログ
  - 商品一覧ページ
  - 商品詳細ページ

Unit-C: 注文・カート
  - カートに追加
  - 注文確定
  - 注文履歴

Unit-D: 支払い（独立性が高い）
  - 支払い処理

Unit-E: 通知
  - メール通知（注文確認・発送）
```

**判断のポイント**:
- 関連する機能をまとめる（凝集性）
- 別チームが担当できるように分ける（独立性）
- 支払いは外部 API 連携があるので独立させると安全
- 通知は Unit-C/D の完了後でも独立して開発できる

</details>

### Q8: アンチパターンの識別

次の Mob Construction の進め方に含まれる問題点を 2 つ挙げてください。

> 「Unit-C（推薦エンジン全体）を実装しましょう。  
>  今日中に全部できると思うので、AI に一気に生成してもらいます。  
>  できたコードは後でゆっくりレビューします。」

<details>
<summary>▶ 答えを見る</summary>

**問題点 1: Bolt に分解せず Unit をそのまま実装しようとしている（Mega Bolt アンチパターン）**

Unit をそのまま実装しようとすると AI への指示が曖昧になり、生成コードの品質が落ちる。また「今日中に全部」という見積もりは非現実的になりやすい。

Bolt（8 時間以内）に分解してから着手すべき。

**問題点 2: 「後でレビュー」（レビュー後回しアンチパターン）**

AI が生成したコードは Mob Construction の中でリアルタイムにレビューするのが AI-DLC の設計。後回しにすると問題が蓄積し、修正コストが増大する。

Bolt 単位で生成 → 即レビュー → 次の Bolt というサイクルが正しい。

</details>

## フェーズ3クイズ（応用）

### Q9: シナリオ判断

あなたのチームが以下の状況に直面しました。どう対応しますか？

> Mob Elaboration で Intent を定義した後、Week 2 の Mob Construction 中に  
> 「この機能、実は既存の外部 API で実現できることがわかった」と判明した。  
> Unit-B（2週間予定）は不要になるかもしれない。どうすべきか？

<details>
<summary>▶ 答えを見る</summary>

**正しい対応**: Intent と Unit を更新し、外部 API 活用に設計を変更する

1. **発見を Mob に共有**: 「Unit-B が外部 API で代替できる」という事実を全員で確認
2. **Intent の更新**: スコープ外に「Unit-B は外部 API を使用する」と記録
3. **Unit-B を縮小**: 「外部 API 統合と動作確認」に変更（2週間 → 2〜3日）
4. **ADR を作成**: 「なぜ自前実装をやめて外部 API にしたか」を記録
5. **空いたリソースの活用**: Unit-C や品質改善に投資

**学習ポイント**: Intent・Unit は「変えてはいけないもの」ではない。発見に応じて更新することが AI-DLC の柔軟性。変更 = 失敗ではなく学習。

</details>

### Q10: 統合問題

以下の Intent からはじめて、Unit に分解し、最初の Bolt を設計してください。

> **Intent**: SaaSプロダクトに「利用状況ダッシュボード」機能を追加する。  
> 目標: 月次解約率を 5% → 3% に削減する。  
> 対象: 管理者ユーザー（各企業の IT 担当）  
> 成功指標: ダッシュボード利用率 60%以上、解約前アラート発報率 80% 以上  

<details>
<summary>▶ 答えを見る</summary>

**回答例**:

**Unit 分割案**:
```
Unit-A: 利用状況データ収集
  - ユーザー行動ログの記録
  - イベント集計バッチ

Unit-B: ダッシュボード UI
  - 利用状況の可視化画面
  - フィルタ・期間選択

Unit-C: リスクスコアリング
  - 解約リスクの計算ロジック
  - アラート閾値設定

Unit-D: アラート通知
  - メール・Slack 通知
  - 通知設定管理
```

**最初の Bolt（Unit-A から着手）**:
```
Bolt-A1: イベントログのスキーマ定義（4時間）
  - 記録すべきイベント種別の定義
  - DynamoDB テーブル設計
  - AI へのプロンプト例:
    「以下の利用イベントを追跡するための DynamoDB テーブル設計をしてください:
     ログイン、機能クリック、ファイルアップロード、API 呼び出し。
     時系列での集計と、企業 ID によるフィルタを前提とする設計に。」
```

**採点基準**: Unit が独立しているか / Bolt が 8 時間以内か / 最初の Bolt から価値を出せるか

</details>
