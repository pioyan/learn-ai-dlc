# E: AI へのプロンプト設計基礎

「実践例に書かれているプロンプトはなぜあの書き方なのか」を理解するための原則集。  
コピペで使えるテンプレートも含む。

---

## 1. プロンプト設計の根本原則

AI は **「何を出力すべきか」が明確なほど良い結果を出す**。

```
曖昧な指示        → 曖昧な出力    → 手直しに時間がかかる
具体的な指示      → 有用な出力    → レビューと修正だけで済む
```

AI-DLC のプロンプトはすべてこの原則に基づいて設計されている。

---

## 2. 良いプロンプトの 5 要素

### [1] Context（背景・状況）
「今どういう状況でこの質問をしているか」を伝える

```
❌ 悪い例:
「推薦エンジンを設計してください」

✅ 良い例:
「ECサイトのバックエンド開発チームです。
月間アクティブユーザー100万人のプラットフォームで
クロスセル向けの推薦エンジンを新規開発します。」
```

### [2] Role（AIの役割指定）
AI に演じてほしい専門家の役割を明示する

```
「あなたはシニアソフトウェアアーキテクトです」
「あなたはDomain-Driven Designの専門家です」
「あなたはセキュリティレビュアーです」
```

### [3] Task（具体的な作業内容）
何をしてほしいかを動詞で明確に指定する

```
❌ 「ドメインモデルについて教えてください」（曖昧）
✅ 「以下の要件からDomain Entityを特定し、
   Aggregateの境界を定義してください」（具体的）
```

### [4] Format（出力形式）
どんな形で返答してほしいかを指定する

```
「Markdown の表形式で出力してください」
「Python クラスとして実装してください」
「箇条書きで 5 つ以内にまとめてください」
「Entity ごとに以下の形式で出力:
  - 名前:
  - 属性:
  - 業務ルール:」
```

### [5] Constraints（制約・条件）
使ってはいけないもの、守るべきルールを伝える

```
「既存 API の破壊的変更は禁止です」
「Python 3.11、pytest を使用してください」
「AWS Lambda の制限（メモリ 256MB、タイムアウト 30 秒）を考慮してください」
```

## 3. AI-DLC の各フェーズごとのプロンプトパターン

### Mob Elaboration（要件明確化）フェーズ

#### Intent 明確化プロンプト
```
【Context】
[ビジネス背景を 3〜5 文で記述]

【Role】
あなたは経験豊富なプロダクトマネージャーとソフトウェアアーキテクトです。

【Task】
この Intent に対して、以下を明確にするための質問を 5〜7 個作成してください:
1. ビジネス目標が曖昧な点
2. 対象ユーザーが不明確な点
3. 成功指標（KPI）が定量化されていない点
4. スコープが広すぎる可能性がある点

【Format】
各質問は以下の形式で:
- 質問:
- なぜ重要か（理由）:
- 想定される回答の選択肢（複数）:
```

#### User Story 生成プロンプト
```
【Context】
以下の Intent ドキュメントをもとに User Story を生成します:
[Intent ドキュメントを貼り付け]

【Role】
あなたは Agile の User Story 作成の専門家です。

【Task】
主要なユーザーペルソナごとに User Story を作成してください。
各 Story には受け入れ基準（Acceptance Criteria）を含めること。

【Format】
# [ユーザーペルソナ名]

## Story [番号]: [タイトル]
As a [ペルソナ],
I want to [目的],
So that [ビジネス価値].

### 受け入れ基準
- [ ] Given [前提条件] When [操作] Then [期待結果]

【Constraints】
- 1 Story の粒度は 1〜3 日で実装できる範囲
- 技術的な実装詳細は含めない（"何を"だけ、"どうやって"は含めない）
```

---

### Mob Construction（設計）フェーズ

#### Domain Model 生成プロンプト
```
【Context】
以下の User Stories と業務要件から Domain Model を設計します:
[User Stories を貼り付け]

【Role】
あなたは Domain-Driven Design（DDD）の専門家です。

【Task】
以下を特定・定義してください:
1. Entity（一意の識別子を持つオブジェクト）
2. Value Object（値が意味を持つ不変オブジェクト）
3. Aggregate（一貫性の境界）
4. Domain Event（重要なビジネスイベント）
5. Repository インターフェース
6. Ubiquitous Language（ドメイン用語集）

【Format】
各 Entity/Value Object は以下の形式:
## [名前]（Entity/Value Object）
- 説明:
- 主要属性: [属性名: 型]
- 業務ルール:
- 関連: [他の Entity/VO との関係]

【Constraints】
- 技術的な実装（データベース設計など）は含めない
- ドメイン言語（業務用語）を使う
```

#### コード生成プロンプト
```
【Context】
以下の Domain Model と技術要件をもとに実装します:
[Domain Model を貼り付け]

技術スタック:
- 言語: Python 3.11
- フレームワーク: AWS Lambda
- データベース: Amazon DynamoDB
- テスト: pytest, moto

【Role】
あなたはシニア Python エンジニアです。

【Task】
[特定の Bolt（例: Bolt-C1: DynamoDB スキーマ設計）] を実装してください。

【Format】
1. 実装コード（型ヒント・docstring 付き）
2. 対応するユニットテスト
3. 設計の判断根拠（なぜこの実装にしたか）

【Constraints】
- OWASP Top 10 のセキュリティ要件を満たすこと
- エラーハンドリングを含めること
- AWS Lambda のメモリ・タイムアウト制限を考慮すること
- 環境変数で設定値を管理すること（ハードコーディング禁止）
```

### コードレビュープロンプト

```
【Context】
以下のコードをレビューしてください:
```python
[コードを貼り付け]
```

【Role】
あなたはシニアソフトウェアエンジニアとセキュリティレビュアーです。

【Task】
以下の観点でレビューしてください:
1. 正確性: 業務ロジックが仕様通りか
2. セキュリティ: OWASP Top 10 の観点で問題がないか
3. パフォーマンス: N+1 問題・非効率な処理はないか
4. 可読性: 変数名・関数名が意図を表しているか
5. テスタビリティ: テストしやすい構造か

【Format】
各問題は:
- 重大度: 🔴 Critical / 🟡 Warning / 🟢 Suggestion
- 場所: [行番号または関数名]
- 問題:
- 推奨修正:
```

## 4. プロンプトを改善するテクニック

### ① 制約を増やして精度を上げる

```
Before（曖昧）:
「テストを書いてください」

After（具体的）:
「pytest を使って、以下の関数のユニットテストを書いてください。
 - 正常系（Success case）2 パターン
 - 異常系（Edge case）3 パターン
 - DynamoDB は moto でモックすること
 - テストデータは fixture で管理すること」
```

### ② 段階的に深める（Chain of Thought）

1つのプロンプトで全部聞かず、段階を踏む:

```
Step 1: 「このシステムで考えられる Domain Entity を列挙してください」
   ↓ チームで確認・修正
Step 2: 「先ほどの Entity をもとに Aggregate の境界を定義してください」
   ↓ チームで確認・修正
Step 3: 「この Aggregate の Python 実装を生成してください」
```

### ③ AI の出力をそのまま使わない

AI の出力は常に「出発点」。チームで以下を確認:
- ドメイン知識として正確か？
- チームの技術スタックに合っているか？
- セキュリティ上の問題はないか？

---

## 5. プロンプトの品質チェックリスト

プロンプトを送る前に確認:

- [ ] Context（背景）を 3 文以上で記述しているか
- [ ] AI の役割（Role）を明示しているか
- [ ] 何を出力してほしいか（Task）が動詞で明確か
- [ ] 出力形式（Format）を指定しているか
- [ ] 使ってはいけない制約（Constraints）を記述したか
- [ ] 関連するコード・ドキュメントを貼り付けたか（Context Engineering）

---

## 6. Context Engineering とは

単なるプロンプトではなく、**AI が最適な出力をするために必要な情報を設計する**こと。

```
良くない例:
「OrderService のテストを書いて」

Context Engineering の例:
「以下のコードベースのコンテキストをもとに OrderService のテストを書いて:

[order_domain.py の全コード]
[order_service.py の全コード]
[既存のテストの例（test_payment_service.py）]

技術スタック: pytest, moto
テスト対象: create_order() メソッド
テストすべきケース: 正常系, DynamoDB 接続失敗, 不正入力」
```

**ポイント**: コードを貼り付けることで AI が一貫したスタイルとパターンでテストを生成できる

---

## 7. まとめ：プロンプト設計の心構え

```
AI は賢い新人エンジニアのようなもの:
- 背景を説明しないと的外れな実装をする
- 「あれ」「それ」では伝わらない
- 良い出力を出すのは指示する側の責任
- 出力をそのまま信じてはいけない（レビューが必須）
```
